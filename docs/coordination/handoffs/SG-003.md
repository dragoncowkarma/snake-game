**발견 사항 (비차단)**

1. **[경계 명확화 권고 — SG-007 착수 전] pause/resume 도메인 진입점 미지정.** 계약은 공개 함수를 `reset`/`enqueueDirection`/`step` 3개로 고정하고 `start`/`restart`/`returnToMenu`/`selectDifficulty`를 `reset`에 매핑하지만, `pause`→`paused`와 `resume`→`playing`(정책 `accept`) 전이는 대응 도메인 함수도, 결과 `GameState`를 누가 생성하는지에 대한 명시도 없다. §7은 동작만 기술한다. 이대로면 SG-007/SG-012가 미문서화 함수를 추가하거나 셸에서 `GameState`를 직접 구성해야 하며, 후자는 §9의 "상태 전이는 domain 소유" 경계와 어색하게 충돌한다. 새 제품 의미가 없으므로 §10상 새 ADR 없이, §1/§4에 `pause`/`resume`(또는 일반 `applyCommand`)를 선언하거나 "pause/resume은 phase+queue만 바꾸는 순수 셸 전이"임을 한 문장으로 못박아 해소 권고.
2. **[사소 — 명확화] `start`/`restart`의 난이도 출처 미명시.** `Command`의 `start`/`restart`는 difficulty payload가 없으나 대응 `reset`은 difficulty를 요구한다. 자연스러운 해석은 `state.difficulty`(모든 phase가 보유)지만 라우터가 이를 전달한다는 문장이 없어 추측 여지가 남는다. 한 문장 보강 권고.
3. **[사소 — 명확화] `PHASE_COMMAND_POLICY`의 `accept` 다의성.** `toggleMute`의 `accept`는 "셸이 처리, `GameState` 불변"인 반면 `start`/`pause` 등의 `accept`는 상태 전이를 뜻한다. §1 산문에는 있으나 기계 판독 상수만 보는 소비자는 mute-accept가 snapshot을 바꾸지 않음을 알 수 없다. 정책 정의부 주석 보강 권고.
4. **[관찰 — plan 범위, 계약 결함 아님] READY→MENU 부재.** `ready.returnToMenu: 'ignore'`는 `DEVELOPMENT_PLAN.md` §3.3 상태도(READY→MENU 간선 없음)를 충실히 반영한다. 다만 READY에서 난이도 변경을 위해 MENU로 돌아갈 경로가 없다. 계약 결함이 아니라 기획/UX 검토 항목으로 상위에 이월.
