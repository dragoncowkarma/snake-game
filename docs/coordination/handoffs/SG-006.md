## 시작 전 확인 사항과 세션 내 승인 경위

작업 시작 시 프롬프트의 `{base_sha}`, `{approved_by}`, `{approved_at}`, `{forbidden_paths}`,
`{acceptance_ids}`, `{required_commands}`가 모두 미채움 placeholder였고, `docs/coordination/
tasks/SG-006.yaml`도 존재하지 않았다. `STATUS.md`는 "SG-006/007/008/009는 새 packet에 최신
base SHA와 허용/금지 경로, 수용 기준, 실제 검증 명령을 담고 claim한 뒤에만 `ready`가 된다"고
명시하고, `AGENTS.md` 3절은 승인 필드가 없으면 시작하지 말라고 규정한다. 이에 따라 임의로
값을 채우지 않고 사람에게 두 차례 직접 질문했다.

1. **승인 방식**: "지금 직접 승인" 선택 — 사람이 이 세션에서 도출한 base_sha/허용/금지 경로로
   즉시 직접 승인했고, 정식 Codex-integrator 작성 packet을 별도로 기다리지 않았다.
2. **구현 범위**: "프롬프트 전체 범위" 선택 — `docs/TASKS.md`의 SG-006 행(정적 shell만)을 넘어,
   같은 문서의 SG-013 행이 기술하는 focus 전이/aria-live/Command 라우팅까지 이번 SG-006에서
   구현하도록 범위를 확장 승인받았다. `docs/TASKS.md`는 이 packet의 허용 경로 밖이므로 본문을
   직접 수정하지 않았고, 이 확장 범위와 TASKS.md/SG-013의 경계 재조정은 조정 책임자의 후속
   과제로 `docs/coordination/tasks/SG-006.yaml`의 `reconciliation_note`에 남겼다.
3. **테스트 경계 예외**: 구현 중 `index.html`을 실제 shell로 바꾸면 `tests/e2e/scaffold.spec.ts`
   (Antigravity 소유, 허용 경로 밖)가 SG-005 placeholder marker 텍스트를 검증하다 필연적으로
   깨진다는 것을 발견해 다시 질문했다. "이번만 scaffold.spec.ts도 수정 허용"을 선택받아, 이
   파일의 단언만 새 실제 menu 화면에 맞게 최소 수정했다(§변경 파일 참고).

세 결정 모두 `docs/coordination/tasks/SG-006.yaml`의 `approval_basis`/`scope_boundary`/
`reconciliation_note` 필드에 근거와 함께 기록했다.

## 결정과 위험

- 새 의존성: 없음. `package.json`은 이 packet의 허용 경로 밖이라 애초에 수정하지 않았고,
  이에 맞춰 DOM 단위 테스트는 jsdom 없이 순수 함수만 분리해 Vitest 기본 `node` 환경으로
  검증했다(§수행 내용). 실제 DOM 렌더/포커스 검증은 기존 Playwright(이미 devDependency)로
  대체했다.
- 새 결정 또는 기존 결정과의 차이:
  - `src/ui/contracts.ts`는 `CONTRACTS.md` §1을 타입만 그대로 미러한 임시 파일이다. SG-007이
    실제 `src/domain/**`를 만들면 이 파일을 지우고 그쪽을 import해야 한다(두 소스 드리프트
    위험, 아래 참고).
  - `index.html`의 스크립트 대상을 `/src/main.ts`(허용 경로 밖)에서 `/src/ui/main.ts`(허용
    경로 안)로 바꿨다. `src/main.ts`/`src/main.test.ts` 자체는 건드리지 않아 지금은
    프로덕션에서 참조되지 않는 orphan 코드로 남아 있다 — 삭제는 이 packet의 허용 경로 밖이라
    하지 않았다. 실제 게임 통합(SG-014) 또는 별도 정리 packet에서 제거를 권장한다.
  - 초기 menu snapshot의 기본 난이도를 `'normal'`로 선택했다(계약에 없는 UI 전용 가정). 선택
    라디오 표시 외 제품 의미는 없다.
  - Restart 커맨드 이후 focus 대상은 QA_PLAN.md DP-B11 절차가 명시적으로 unspecified라고
    적어 두었다. Start와 동일하게 "ready 진입 시 board"로 통일했다(문서화된 선택, 근거는
    `state.ts`의 `focusTargetForTransition` 주석).
  - `dispatch`는 아직 프로덕션에서 콘솔 로그만 남기는 스텁이다. `toggleMute`만은 계약대로
    shell이 로컬로 시각 상태를 바꾸고 동시에 `dispatch`도 호출한다.
- 알려진 위험/제약:
  - **타입 드리프트**: `src/ui/contracts.ts`가 `CONTRACTS.md`와 수동으로 동기화된 상태다.
    SG-007 완료 후 즉시 교체하지 않으면 두 정의가 벌어질 수 있다.
  - **CSS `[hidden]` 우선순위 함정**: 개발 중 `.panel { display: flex }`가 `[hidden]`의
    UA `display:none`을 덮어써 모든 phase 패널이 동시에 보이는 실제 버그를 발견해
    `.panel:not([hidden])`+전역 `[hidden] { display: none !important; }`로 고쳤다. 이후
    다른 CSS에서 `display`를 직접 지정하는 선택자를 추가할 때 같은 함정에 유의해야 한다.
  - **대비 회귀 위험**: 최초 border 토큰이 3:1 미달이었다(§AC08). 디자인 토큰을 조정하는
    후속 작업은 반드시 대비를 재계산해야 한다(§리뷰어 재현 절차 4에 재현 스니펫 포함).
  - `src/main.ts`/`main.test.ts` orphan 파일이 남아 있어 다음 담당자가 실수로 다시 참조하지
    않도록 주의해야 한다.
